\subsection{How Tests Were Executed (Aligned with the V\&V Plan)}
The following evidence types were produced exactly as described in the V\&V Plan:
\begin{itemize}
  \item \textbf{Manual functional tests (FR):} Executed by the team using defined step-by-step procedures in the V\&V Plan and validated through observation of expected system behavior.

\section{Comparison to Existing Implementation}

Prior to this system, reconstruction of manuscript fragments was primarily performed manually by scholars using static image collections. Researchers would visually inspect fragments, manually compare edges and shapes, and attempt to infer relationships between fragments based on visual similarity and domain knowledge. This process is time-consuming, difficult to reproduce, and does not scale well to large fragment collections.

The \textit{Sanskrit Ciphers} platform improves upon this workflow by integrating computational tools that support fragment analysis and reconstruction. Compared to the traditional approach, the implemented system provides the following improvements:

\begin{itemize}
\item \textbf{Automated fragment enrichment:} Machine learning models automatically analyze fragments and generate metadata such as segmentation masks, script type, line count, circle detection, and scale estimation. This reduces the manual effort required to characterize fragments.

\item \textbf{Search and filtering capabilities:} Users can quickly locate fragments based on metadata attributes, which is not possible in static image collections.

\item \textbf{Interactive reconstruction workspace:} The canvas interface allows fragments to be visually arranged, rotated, and compared during reconstruction sessions, improving usability compared to manual inspection.

\item \textbf{Reproducible computational pipeline:} The ML pipeline processes fragments in a consistent and repeatable manner, allowing results to be reproduced and verified.

\item \textbf{Scalability for large fragment datasets:} Automated analysis enables the system to process large fragment collections more efficiently than manual methods.
\end{itemize}

Overall, the implemented system extends existing manual reconstruction workflows by providing automated analysis, searchable metadata, and an interactive reconstruction interface. The improvements verified in this report demonstrate that the system successfully supports the reconstruction workflow described in the V\&V Plan.

\begin{enumerate}
  \item What went well while writing this deliverable?

  We had artifacts such as verification scripts, configuration files, and integration checklists, which made it possible to connect implemented functionality to explicit V\&V evidence easily.

  \item What pain points did you experience during this deliverable, and how
    did you resolve them?

  The largest pain point was uneven availability of test results across all requirement areas. We addressed this by separating evidence into executed checks, verification based on artifacts, and planned tests.

  \item Which parts of this document stemmed from speaking to your client(s) or

  Requirement priorities around usability, reconstruction workflows, and transparency of ML outputs reflect our advisor's priorities. Implementation validation on a lower level such as processor registration and config correctness do not stem from discussion with our advisor.

  \item In what ways was the Verification and Validation (VnV) Plan different
  from the activities that were actually conducted for VnV? If there were
  differences, what changes required the modification in the plan? Why did
  these changes occur? Would you be able to anticipate these changes in future
  projects? If there weren't any differences, how was your team able to clearly
  predict a feasible amount of effort and the right tasks needed to build the
  evidence that demonstrates the required quality? (It is expected that most
  teams will have had to deviate from their original VnV Plan.)

  The plan defined broad FR and NFR test suites, including load, reliability, and usability studies. In reality, a larger portion of completed evidence became dependent on integration for active ML features. This was caused by time and environment constraints that usually occur in ML projects due to compute requirements and feature scope. In future projects, we will create milestones for producing result artifacts such as performance logs, usability summaries, and coverage reports earlier, so final reporting has more depth across all requirement categories.
\end{enumerate}
